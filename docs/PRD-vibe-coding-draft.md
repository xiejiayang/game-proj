## 1. 产品定位

### 1.1 一句话描述
一款以都江堰治水智慧为主题的**网页单机解谜游戏**，玩家通过摆放治水构件来引导水流、保护村庄，在“筑墙-失败-顿悟”的循环中理解“堵不如疏、顺势而为”。

### 1.2 定位策略（已确定：方案 C）
- **主体定位**：泛网页文化解谜游戏（都江堰主题）。
- **未来扩展**：景区现场获客、AR 实景终章作为 v2.0/v3.0 的可选扩展。
- **商业模式**：免费 + 广告（广告系统 v1.0 再接入，切片阶段仅预留广告位）。

### 1.3 平台规划
| 阶段 | 平台 | 说明 |
|------|------|------|
| Vertical Slice | Web（PC/手机浏览器） | 1 关完整体验，验证核心循环 |
| v1.0 | Web + 微信小程序小游戏 | 8 关完整版，接微信登录/云存档/广告 |
| v2.0+ | 景区触屏 / AR | 可选扩展，依赖外部合作 |

### 1.4 与原始立项文档的差异
- 原方案：Unity 原生 App + 微信小游戏，景区获客优先。
- 本方案：Unity WebGL 网页优先，泛用户验证，景区/AR 后置。

## 2. 用户痛点

### 2.1 第一目标用户
**文化爱好者 × 休闲解谜玩家**的交集。

### 2.2 核心痛点
玩家想玩到**“机制即知识”**的文化解谜游戏，但市面上的产品：
- 要么太说教（像互动 PPT）；
- 要么只是文化换皮（都江堰换成其他主题也不影响玩法）。

### 2.3 需求验证标准
- 玩家通关后能用自己的话解释“堵不如疏”；
- 玩家愿意主动重试以寻找更节俭的解法；
- 玩家愿意分享“我懂了李冰”的成就感。

## 3. MVP 范围

### 3.1 Vertical Slice 目标
只做 **L1《堵》** 一关，但按最终品质完成：完整的 3D 水墨表现、核心循环、状态机、结算、碑廊解锁。

### 3.2 包含内容
- [ ] L1 关卡配置：地形、水源、预置官方石墙、村庄、竹笼/杩槎库存。
- [ ] 编辑/模拟/结算/暂停状态机。
- [ ] 规则粒子水流模拟（石墙反弹、竹笼分流）。
- [ ] 资源系统（料/工/时）与节俭判定。
- [ ] 撤销/重置/重试。
- [ ] 老河工分层提示系统（L1 仅开放前两层）。
- [ ] 结算弹窗 + 碑廊解锁（两屏弹窗）。
- [ ] 本地存档（PlayerPrefs / localStorage）。
- [ ] 设置页（音量、画质、语言预留）。

### 3.3 明确不包含（Deferred）
- L2–L8 关卡。
- 完整碑廊系统（仅 L1 通关条目）。
- 广告系统（仅预留 Banner 位）。
- 微信登录/云存档。
- 完整音效配音（可用占位音效）。
- L8 情感终章/AR。

### 3.4 切片退出标准（Exit Criteria）
1. 玩家能在 5–10 分钟内完成 L1 并理解“堵不如疏”。
2. WebGL 在主流手机和 PC 浏览器上稳定 30fps 以上。
3. 首屏加载时间 < 15 秒（4G 网络）。
4. 水墨风格可被识别（不一定完美，但看得出是水墨）。

## 5. 核心流程

### 5.1 L1 完整玩家旅程

```
启动/标题页
    │
    ▼
L1 引子动画（可跳过）
    │
    ▼
编辑阶段（Editing）
    │  ├─ 河道中央预置官方石墙
    │  ├─ 底部工具栏：竹笼、杩槎
    │  ├─ 拖放/旋转/撤销/重置
    │  └─ 实时显示料/工/时消耗
    │
    ▼
点击「放水」
    │
    ▼
模拟阶段（Simulating）
    │  ├─ 水源发射水粒子
    │  ├─ 石墙被冲毁或竹笼分流
    │  └─ 统计村庄受击粒子数
    │
    ▼
结算阶段（Settling）
    │  ├─ 失败：回到编辑阶段，保留提示触发计数
    │  └─ 成功：弹出「暂时安全」+ 碑廊解锁
    │
    ▼
结算弹窗
    │  ├─ 屏1：「暂时安全」/「墙倒了」/「村子仍被淹」
    │  └─ 屏2：碑廊「堵不如疏」图文
    │
    ▼
结束选择：「再试一次」 / 「下一关（占位）」
```

### 5.2 提示触发规则
- 第 1–2 次失败：无提示，让玩家自己试错。
- 第 3 次失败：自动触发老河工第一层提示（描述现象）。
- 第 5 次失败：触发第二层提示（指向工具）。
- 切片阶段不开放第三层（接近解法）。

### 5.3 状态机
```
Editing ──[点击放水/布局合法]──> Simulating
Simulating ──[时间到/触发条件]──> Settling
Settling ──[点击重试]──> Editing
Settling ──[点击下一关]──> 关卡系统（占位）
Simulating ──[点击暂停]──> Paused
Paused ──[点击继续]──> Simulating
```

### 5.4 成功/失败判定
- **成功**：模拟时长 ≥ 关卡最短时长，且结算时村庄未被淹没。
- **失败原因**：
  - `flood`：村庄受击粒子累计 ≥ 阈值。
  - `destroyed`：关键构件损毁（L1 为预置石墙倒塌，非玩家失败条件，仅叙事）。
  - `timeout`：模拟超时。
- **节俭判定**：成功且实际消耗 ≤ 节俭阈值。

## 6. 技术方案

### 6.1 引擎与版本
- **引擎**：Unity 6.3 LTS
- **渲染管线**：URP（Universal Render Pipeline）
- **导出目标**：WebGL（网页），后续用 Unity 微信小游戏转换工具导出微信小游戏

### 6.2 架构边界
- **Unity 全包**：3D 场景、Gameplay、HUD、所有页面弹窗、设置、存档均在 Unity 内。
- **外部网页**：仅做加载壳、全屏按钮、WebGL 不支持提示。

### 6.3 水流模拟
- **方案**：规则粒子系统。
- **固定时间步长**：例如 0.02s，与渲染帧率解耦，保证确定性。
- **L1 支持的 WaterInteraction**：
  - 石墙：反弹 + 耐久伤害。
  - 竹笼/杩槎：按法线方向分流。
- **粒子视觉**：Trail Renderer + 水墨 Shader，速度越快墨色越浓。

### 6.4 3D 水墨实现路径
| 层级 | 实现方式 | 切片阶段 |
|------|----------|----------|
| 场景基底 | 低多边形 3D 地形 + 水面片，淡墨色调材质 | ✅ |
| 水墨材质 | URP Shader Graph：宣纸纹理、边缘晕染、墨色渐变 | ✅ |
| 后处理 | 全局水墨勾边、纸质噪点、远景雾化 | ✅ 尽量 |
| 人物/村民 | 2D 剪影或 AI 生成水墨立绘 | ✅ |
| 实时水墨流体 | 水流本身做成水墨笔触 | ❌ Deferred |
| 全 3D 角色建模 | 李冰、村民等精细模型 | ❌ Deferred |

### 6.5 存档系统
- **切片阶段**：Unity PlayerPrefs（浏览器 localStorage）。
- **v1.0**：本地存档为主，可选微信云存档。
- **Deferred**：Java 后端仅用于 v1.0 后的数据分析和可选跨设备同步。

### 6.6 微信小游戏适配约束（提前遵守）
- 控制首包资源，切片目标 < 10MB。
- 避免全屏重后处理（如全屏模糊、复杂屏幕空间反射）。
- UI 同时支持鼠标和触摸。
- v1.0 使用 AssetBundle + CDN 按需加载关卡资源。

### 6.7 美术资产策略
- **3D 场景**：Asset Store 低多边形山水/村庄包 + 自定义淡墨材质。
- **核心构件**：竹笼、杩槎、石墙用简单几何体或可商用模型。
- **水流/粒子**：程序化生成 + Trail Renderer + Shader。
- **人物/叙事图**：AI 生成水墨风格 2D 图。
- **Shader/后处理**：自研 Shader Graph。

## 7. 主要风险与缓解

### 7.1 Top 3 风险（用户指定）

#### 风险 1：3D 水墨 + WebGL 性能做不出来
- **缓解**：
  - 先用灰盒材质跑通 L1 逻辑，确认帧率后再上水墨 Shader。
  - 模型保持低多边形，复杂水墨感由 Shader/后处理承担。
  - 粒子数量可控，必要时动态降质。

#### 风险 2：Unity WebGL 加载时间劝退
- **缓解**：
  - 切片目标首包 < 10MB。
  - 使用 LZMA/Brotli 压缩。
  - 加载页做进度条 + 水墨动画，降低等待焦虑。
  - v1.0 用 AssetBundle 延迟加载非当前关卡资源。

#### 风险 3：美术资产做不出水墨感
- **缓解**：
  - 水墨感 70% 靠 Shader/后处理，模型只负责轮廓。
  - 人物用 2D 剪影/AI 生成图，避开 3D 人物建模。
  - 先做 mood board 和概念图，确认风格方向再量产资产。

### 7.2 其他风险

#### 风险 4：核心循环不好玩
- **缓解**：编码前先纸面原型测试 L1 布局；找 3–5 人试玩验证顿悟感。

#### 风险 5：范围蔓延
- **缓解**：严格执行切片 Exit Criteria，任何超出 L1 的需求必须走变更记录。

#### 风险 6：微信小游戏转换踩坑
- **缓解**：切片阶段即查微信小游戏不支持功能清单，避开 WebGL 扩展和特定插件。

#### 风险 7：广告变现收益低
- **缓解**：切片阶段不接广告，先验证留存；v1.0 再评估变现模式。
